No new features, no new pages, no Python logic changes ‚Äî CSS and template markup only.

<decisions>
## Implementation Decisions

### Breadcrumb & navigation
- FlowSurgeon logo in topbar is ALWAYS a link to home (`/flowsurgeon`), on every page including home
- On detail pages: logo is the brand/home link, breadcrumb starts at route level ‚Äî NO duplicate "FlowSurgeon" text in breadcrumb
- Detail page breadcrumb structure: `FlowSurgeon [logo]  /  [GET] /books  /  abc123` ‚Äî three levels
- Route detail page (filtered requests list) breadcrumb: `FlowSurgeon [logo]  /  [GET] /books`
- Separator: forward slash `/` (not chevron ‚Ä∫)
- Remove the current `<span class="logo">FlowSurgeon</span>` + `<a class="b-dim">FlowSurgeon</a>` duplicate ‚Äî replace with single clickable logo + route-level breadcrumb

### Filter controls (method filter)
- Redesign from loose pill buttons to shadcn-style toggle group: single bordered container with segmented buttons inside
- Active item: subtle surface fill (`--bg-elevated` or equivalent) + white/bold text ‚Äî NOT brand-green
- Inactive items: ghost style (transparent bg, muted text)
- Position: content area (above routes list), NOT in topbar
- Route count ("12 routes") right-aligned on the same row as the toggle group
- Remove the current `.pill` / `.pill-active` / `.filter-pills` classes, replace with toggle group pattern

### Visual weight & animations
- REMOVE all entrance animations: `fadeIn`, `cardIn`, `slideDown` keyframes and all `animation:` properties that reference them
- KEEP subtle hover/transition effects on interactive elements (buttons, rows, tabs) ‚Äî `transition: var(--transition-base)` stays
- Swagger-tight row density: ~36-40px row height for route list rows (currently much taller)
- Tighten overall page padding: reduce from `--space-7`/`--space-8` (32px/48px) to ~16-20px for content area margins
- Reduce topbar height if currently oversized (64px may be fine, evaluate against Swagger reference)
- The goal: dense, scannable, professional ‚Äî like Swagger UI's endpoint list

### Light mode
- Triggered by `prefers-color-scheme: light` media query only ‚Äî NO manual toggle in UI
- Implementation: override CSS custom properties in `@media (prefers-color-scheme: light) { :root { ... } }`
- Light mode reference: Swagger UI light theme (white background, light gray surfaces, dark text)
- Method badges: SAME Swagger colors in both modes (blue GET, green POST, etc.) with white text ‚Äî proven to work on both light and dark backgrounds
- Status colors (2xx/4xx/5xx): same hues but adjusted for proper contrast on light backgrounds ‚Äî slightly darker/more saturated foreground, lighter pastel backgrounds
- Duration colors: same adjustment as status colors
- Border color: light gray (not dark gray) ‚Äî standard light mode border
- Text hierarchy: dark text primary, medium gray for muted/dim ‚Äî inverted from dark mode

### Claude's Discretion
- Exact light mode color values (specific hex codes for light backgrounds, surfaces, borders, text)
- Exact row height value within 36-40px range
- Exact page padding value within 16-20px range
- Toggle group border-radius and internal spacing
- Whether topbar height needs adjusting
- How to handle the toggle group on very narrow viewports (desktop-only tool, but reasonable minimum)

</decisions>

<specifics>
## Specific Ideas

- Swagger UI is THE visual reference for both dark and light mode list views ‚Äî dense, clean, professional
- shadcn/ui ToggleGroup component is the reference for the filter controls
- The vibrant/gaming color palette is INTENTIONAL and wanted ‚Äî do not desaturate for dark mode
- Light mode must handle the same vibrant colors gracefully ‚Äî adjust contrast, not vibrancy
- The user's core complaint: "you tried, but it didn't just work out" ‚Äî spacing and density are the main culprit, not colors
- Remove all translateY animation effects ‚Äî instant rendering feels snappier and more professional

</specifics>
